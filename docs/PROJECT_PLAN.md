# BrandPilot Backend Rebuild - Project Plan

이 문서는 프로젝트가 존재하는 이유, 어떤 방식으로 공부하고 개발할지, 기존 프로젝트의 무엇을 보완할지와 최종 완료 기준을 관리합니다. 일상적인 진행 상황은 이 문서가 아니라 `PROJECT_STATE.md`에 기록합니다.

## 1. 프로젝트를 진행하는 목적

기존 BrandPilot 개발을 마지막으로 약 6개월 동안 백엔드 개발을 쉬었습니다. 그동안 DTO가 무엇인지, 요청과 응답 객체를 어떻게 작성하는지부터 다시 확인해야 할 정도로 Spring 백엔드 개발 감각이 떨어졌습니다.

이 프로젝트의 첫 번째 목적은 완성 속도가 아니라 **백엔드 기본기를 작은 단위로 다시 익히고 직접 코드로 작성하면서 개발 감각을 회복하는 것**입니다.

두 번째 목적은 **기존 BrandPilot에서 아쉬웠던 설계와 품질 문제를 보완해 같은 서비스를 더 신뢰할 수 있는 백엔드로 다시 만드는 것**입니다.

세 번째 목적은 **작성한 API, 데이터 모델, 인증, 트랜잭션과 테스트를 면접에서 자신의 언어로 설명할 수 있게 되는 것**입니다.

## 2. 제품 목표

BrandPilot은 브랜드 구축 경험이 부족한 예비 창업자와 초기 기업 담당자가 단계별 질문에 답하고, 생성된 후보를 선택하여 이름·콘셉트·스토리·로고로 구성된 하나의 브랜드를 완성하고 관리할 수 있게 하는 백엔드 서비스입니다.

핵심 흐름:

```text
회원가입
→ 로그인
→ 진단 제출과 Brand 생성
→ 네이밍 후보 생성·선택
→ 콘셉트 후보 생성·선택
→ 스토리 후보 생성·선택
→ 로고 후보 생성·선택
→ 최종 결과 조회
```

사용자는 중간에 중단해도 저장된 현재 단계부터 이어서 진행할 수 있어야 합니다.

## 3. 학습 중심 개발 방식

### 기본 원칙

- 사용자가 구현 코드를 직접 타이핑합니다.
- Codex는 완성 기능 전체를 대신 작성하지 않습니다.
- 모르는 개념은 구현 직전에 기초부터 설명합니다.
- 한 번에 한 개의 `TODO`만 구현합니다.
- 사용자가 작성한 코드는 Codex가 먼저 읽고, 바로 고치지 않고 문제와 이유를 설명합니다.
- 동작 확인만으로 완료하지 않고 테스트와 설명까지 끝내야 합니다.

### 하나의 기능을 만드는 순서

기능은 Controller만 만들거나 Service만 만드는 단위가 아니라, 사용자가 실제로 호출할 수 있는 작은 수직 기능 단위로 진행합니다.

```text
1. 현재 체크포인트와 Git 상태 확인
2. 관련 API 계약과 ERD 확인
3. 이번 기능에서 공부할 개념 확인
4. 전체 구현 순서를 번호가 있는 TODO로 작성
5. TODO 1의 목적과 작은 예제 설명
6. 사용자가 직접 코드 입력
7. Codex 코드 리뷰와 개념 설명
8. 좁은 단위 테스트 실행
9. 다음 TODO 반복
10. HTTP·DB 통합 흐름 검증
11. 면접에서 설명할 핵심을 말로 정리
12. PROJECT_STATE 갱신
13. 커밋 체크포인트 판단 후 Git push
```

### TODO 작성 규칙

- 새 클래스나 기능을 시작할 때 전체 순서를 한국어 `TODO 1`, `TODO 2` 형식으로 먼저 보여줍니다.
- Codex는 필요한 패키지, 클래스와 메서드 뼈대까지만 준비할 수 있습니다.
- 아직 사용자가 작성하지 않은 구현을 import나 완성 메서드로 미리 숨겨두지 않습니다.
- 한 TODO가 검토와 테스트를 통과하기 전 다음 TODO로 넘어가지 않습니다.
- 완료된 TODO 주석은 코드에 계속 쌓아두지 않고, 사용자가 개념을 설명할 수 있으면 제거합니다.

## 4. 기존 프로젝트에서 유지할 장점

### 단계형 도메인

진단부터 네이밍, 콘셉트, 스토리, 로고와 최종 결과로 이어지는 흐름에는 상태 전이와 순서 제약이라는 명확한 도메인 규칙이 있습니다. 단순 CRUD보다 백엔드 설계를 학습하기 좋은 핵심입니다.

### 사용자 소유권 검증

인증 사용자가 자신의 브랜드만 조회·변경·삭제할 수 있게 했던 방향을 유지합니다. 리빌드에서는 인증과 소유권 검증을 명확히 구분하고 테스트합니다.

### 생성 후보와 최종 선택의 분리

재생성 가능한 후보와 사용자가 확정한 결과를 구분하는 방향을 유지합니다. 리빌드에서는 생성 회차, 활성 후보와 최종 선택의 의미를 관계형 데이터로 명시합니다.

### 외부 시스템 경계 경험

기존 프로젝트에서 AI 서버, S3, Docker, Nginx와 Kafka를 연결한 경험은 유지합니다. 이번에는 기술 개수보다 교체 가능한 경계와 실패 처리를 정확히 구현합니다.

## 5. 기존 프로젝트의 문제와 보완 목표

### 1. 요구사항과 용어가 코드보다 먼저 정리되지 않음

- 기존 문제: 서비스 방향 변경의 흔적과 서로 다른 용어가 계층마다 남았습니다.
- 보완 목표: API, DTO, Entity와 테이블에서 같은 개념에 같은 이름을 사용하고 핵심 범위를 먼저 확정합니다.

### 2. 공개 API가 `Map<String, Object>`에 의존

- 기존 문제: 타입, 필수값과 길이 검증이 불명확하고 런타임 오류 가능성이 컸습니다.
- 보완 목표: 유스케이스별 고정 DTO와 Bean Validation을 사용하고 API 계약과 테스트를 일치시킵니다.

### 3. 공통 오류 응답과 HTTP 상태 코드가 불명확

- 기존 문제: 입력 오류, 권한 오류와 단계 충돌이 일반 예외로 전파되어 500이 될 수 있었습니다.
- 보완 목표: 오류 코드, 도메인 예외와 `@RestControllerAdvice`를 사용해 400·401·403·404·409·500을 구분합니다.

### 4. Security 설정과 서비스 코드의 가정이 다름

- 기존 문제: 공개 경로 설정과 Controller의 인증 가정이 어긋났고 JWT를 로그에 출력했습니다.
- 보완 목표: 공개·보호 API를 명확히 나누고 401·403을 테스트하며 민감 정보를 기록하지 않습니다.

### 5. 핵심 규칙을 증명하는 테스트 부족

- 기존 문제: 상태 전이, 소유권과 실패 흐름을 수동으로만 확인했습니다.
- 보완 목표: 도메인 단위 테스트, Service 테스트, Controller 테스트와 MySQL 통합 검증을 목적에 맞게 구분합니다.

### 6. 외부 호출이 긴 DB 트랜잭션 안에 존재

- 기존 문제: AI 호출 지연 동안 DB 커넥션과 트랜잭션이 유지될 수 있었습니다.
- 보완 목표: 조회·검증, 외부 후보 생성과 저장을 분리하고 변경에 필요한 트랜잭션만 짧게 유지합니다.

### 7. DB와 파일 저장소 사이의 일관성 부족

- 기존 문제: 파일 저장과 DB 작업 중 하나만 성공하면 고아 파일이나 잘못된 참조가 남을 수 있었습니다.
- 보완 목표: 파일 저장 인터페이스, 보상 삭제, 재시도 정책과 안전한 파일 검증을 구현합니다.

### 8. DB 스키마 버전 관리 부재

- 기존 문제: `ddl-auto: update`에 의존해 변경 이력과 환경 재현이 어려웠습니다.
- 보완 목표: Flyway를 스키마의 기준으로 사용하고 Hibernate는 `validate`만 수행합니다.

### 9. 동시성 고려 부족

- 기존 문제: 같은 Brand의 같은 단계를 동시에 생성하거나 선택할 수 있었습니다.
- 보완 목표: UNIQUE 제약, 낙관적 락과 필요 시 비관적 락을 비교하고 동시성 테스트로 정책을 증명합니다.

### 10. 계층 책임과 의존 방향이 일관되지 않음

- 기존 문제: Controller가 구현 Service에 직접 의존하고 Entity Setter가 넓게 열려 있었습니다.
- 보완 목표: Controller는 HTTP, Service는 유스케이스와 트랜잭션, Entity는 상태 규칙을 담당합니다. 인터페이스는 실제 교체 경계에만 사용합니다.

### 11. 로그와 환경 설정 부족

- 기존 문제: 임시 출력, 민감 정보 로그와 재현하기 어려운 환경 설정이 존재했습니다.
- 보완 목표: SLF4J, 환경변수와 프로필을 사용하고 실행 조건을 문서로 재현합니다.

### Kafka에 대한 결정

Kafka는 서비스 필수 요구사항이 아니라 비동기 처리를 공부하기 위한 실험이었습니다. 핵심 리빌드에서는 제외하고, 재시도·멱등성·DLQ·Consumer Group과 Offset까지 학습할 준비가 되었을 때 별도 실험으로 진행합니다.

## 8. Git 커밋과 Push 규칙

### 커밋할 시점

다음 조건을 모두 만족한 작은 기능 단위에서 커밋을 제안합니다.

- 하나의 설명 가능한 목적을 가집니다.
- 미완성 뼈대가 아니라 의미 있는 동작이 완료되었습니다.
- 관련 빌드나 테스트가 통과했습니다.
- 문서의 체크포인트가 실제 코드와 일치합니다.
- 비밀번호, 토큰, IDE 파일과 빌드 결과가 포함되지 않습니다.

좋은 커밋 예시:

- Flyway migration과 대응 Entity를 MySQL에서 검증
- 회원가입 Service와 정상·실패 단위 테스트 완료
- 회원가입 Controller와 201·400·409 테스트 완료
- 확정된 인증 결정과 API·ERD 동기화

필드 하나, import 하나, 빈 클래스나 실패하는 테스트는 커밋 단위가 아닙니다.

### 커밋 절차

```text
git status와 diff 확인
→ 관련 테스트 실행
→ PROJECT_STATE 갱신
→ 커밋 범위와 제외 파일 확인
→ 사용자가 커밋 여부 결정
→ commit
→ push
→ 다음 체크포인트 시작
```

Codex는 자동으로 커밋하거나 push하지 않습니다. 사용자가 요청한 경우에만 대상 파일을 제한해 수행합니다.

## 9. 최종 완료 기준

### 기능

- 회원가입과 로그인이 동작합니다.
- 인증 사용자가 자신의 Brand만 관리할 수 있습니다.
- 진단부터 로고까지 단계 순서와 중단 후 재개가 보장됩니다.
- 후보 생성·재생성·선택과 영구 삭제 정책이 동작합니다.
- 로컬 파일 저장·조회·삭제와 실패 보상 정책이 동작합니다.

### 품질

- 빈 MySQL에서 Flyway만으로 동일한 스키마를 만들 수 있습니다.
- API 계약, DTO, Entity와 DB 제약조건이 서로 일치합니다.
- 정상 흐름뿐 아니라 검증, 인증, 소유권, 상태 충돌과 동시성 실패가 테스트됩니다.
- 트랜잭션과 외부 의존성 경계를 설명할 수 있습니다.
- Secret과 개인정보가 코드, 문서와 로그에 없습니다.

### 학습·면접

- 작성한 코드를 보지 않고 요청이 Controller부터 DB까지 처리되는 흐름을 설명할 수 있습니다.
- DTO, DI, JPA 영속성 컨텍스트, 트랜잭션과 JWT를 BrandPilot 코드에 연결해 설명할 수 있습니다.
- 주요 설계 선택마다 대안과 트레이드오프를 말할 수 있습니다.
- 기존 프로젝트의 문제와 리빌드에서 어떻게 보완했는지 코드와 테스트를 근거로 설명할 수 있습니다.
- 프로젝트를 30초, 2분과 5분 버전으로 설명할 수 있습니다.
